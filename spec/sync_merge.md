# SPEC · 一体多身 · 多设备同步与整合（契约草案 v0.1）

> 状态：草案，Claude 起草（2026-07-13），待 Codex 复核。
> 依据 v4.0 总体策划 §5.9-5.12（创始人新增需求）。第 6 份契约。

## 目标

同一 Life ID 装在多台电脑/手机上都是"同一个它"；一台更新其他自动同步到最新；两台往不同方向更新时能整合。支持多 Life ID 切换登录。**繁殖 ≠ 升级**：多设备同步是成长（同一 Life ID），不是繁殖。

## 身份与副本

```
LifeReplica = {
  life_id,                 # 全局唯一（= 身份密钥指纹）
  device_id,              # 本副本所在设备
  vector_clock,           # 本副本已知的各设备逻辑时钟
  head,                   # 本副本事件账本的 Merkle 头哈希
}
```
一台设备可持有多个 LifeReplica（不同 life_id），互相隔离（独立密钥、独立加密存储、独立记忆命名空间）。切换登录 = 切换前台 life_id。

## 同步：只增事件账本的并集

- 同步单位是 **LifeEvent**（见 life_event.md），只增不改。每条带 `content_hash + vector_clock + device_id + wall_time`。
- 同步协议：交换 head → 对方缺哪些事件 → 传输（端到端加密）→ 双方各自 append 并集 → **各自确定性重建派生状态**（记忆/索引，不传输派生态）。
- 传输层：优先设备↔设备 P2P；可选用户可控中继（只存密文）；**不强制中央服务器**。
- 收敛保证：最终一致性——任意两副本同步后 head 相同。

## 整合：三层冲突处置（核心）

日志层并集无冲突；**语义冲突**（两处对同一件事矛盾判断）走三层：

1. **自动合并（默认，约 95%）**：不同对象 → 并集。同一对象两处改 → 按 vector_clock 取较新为 current，**保留另一版本为备选，绝不静默丢弃（union over overwrite）**。
2. **代谢态自调和**：真矛盾 → 交给"认知分歧"机制（同调和内部矛盾信念的逻辑：证据/时近/置信度/来源），产出合并理解 + 低置信度打标。
3. **每日三省问主人**：无法自信调和（尤其**价值层**）→ 端给主人拍板，决定回写为镜核训练样本。

## 分类型合并规则

| 数据类型 | 合并规则 |
| --- | --- |
| LifeEvent | 永远并集（无冲突） |
| 事实 MemoryRecord | supersedes 链，vector_clock 定 current，保留分支 |
| 偏好 | 时近 + 置信度 |
| 能力基因 | 成熟度生命周期：更受检验版本胜，或两版并存为 variant |
| 价值层 / 宪章解释 | **永不自动合并，必上抛主人** |
| 天性/宪章（不可变层） | **永不合并**；检测到分歧 = 篡改告警，免疫系统介入 |

每次合并写一条 **MergeReceipt**（who_diverged、fields、resolution、resolver、vector_clocks）。

## 硬性验收（进 TESTS，对应 0.1 F 组）

- [ ] 两台设备同 Life ID，各自更新不同对象 → 同步后双向收敛、head 一致、无丢失。
- [ ] 制造一次同对象语义冲突 → 走完三层整合，保留双分支，价值层冲突上抛每日三省。
- [ ] 天性层被人为改一处 → 同步时触发篡改告警，不被当作普通合并。
- [ ] 多 Life ID 切换登录，副本互相隔离，无串读。
